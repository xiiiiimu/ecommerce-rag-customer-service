# 电商智能客服知识库系统 — 技术文档

> 项目：`knowledge_system`  
> 栈：Spring Boot 3.2 · Java 17 · Spring AI 1.0.1  
> 定位：Reliability-oriented Agentic RAG + 订单/工具融合的智能客服后端

---

## 1. 项目概述

### 1.1 目标

为电商客服场景提供基于知识库的智能问答能力，覆盖：

- 政策/FAQ 类问题（退款、物流、优惠券、大促规则等）
- 订单状态查询（结构化数据）
- 订单 + 规则混合问题（如「发货了吗？能退款吗？」）
- 知识库运维（列文件、查版本、上传任务等）

### 1.2 与普通 RAG 的差异

| 维度 | 普通 RAG | 本项目 |
|------|----------|--------|
| 核心问题 | 怎么找文档、怎么生成 | **该不该检索、能不能答、答不好怎么办** |
| 入口 | 统一检索 + LLM | **编排层**分流 + 多 AnswerMode |
| 幻觉控制 | 主要靠 Prompt | **证据门控**，不足/冲突可不调用 LLM |
| 多轮 | 可选对话记忆 | **会话状态机** + 升级分 + 转人工 |
| 订单/工具 | 常混在 RAG 里 | **确定性路径**，不依赖 LLM |

### 1.3 技术栈

| 类别 | 技术 |
|------|------|
| 框架 | Spring Boot 3.2、Spring AI、MyBatis |
| LLM | DeepSeek `deepseek-chat`（OpenAI 兼容 API） |
| Embedding | Ollama `nomic-embed-text`（768 维） |
| 向量库 | PostgreSQL + pgvector |
| 全文检索 | Elasticsearch 8.x（BM25） |
| 缓存 | Redis、Caffeine |
| 消息队列 | Kafka（异步入库） |
| 分布式锁 | Redisson |
| 文档解析 | Apache Tika |
| 可观测 | 文件日志 `logs/app.log` + `METRIC` 埋点 |

## 3. 核心流程

### 3.1 智能问答主链路（`/chat/ask`）

**请求：**

```json
POST /chat/ask
{
  "sessionId": "user_001",
  "question": "双11优惠券过期后还能用吗"
}
```

**响应：**

```json
{
  "question": "...",
  "answer": "...",
  "chunks": [ { "fileName", "content", "chunkIndex", "score", ... } ]
}
```

**流程（`reliability.enabled=true`）：**

1. 加载 `ConversationState`（Redis `rag:state:{sessionId}`）
2. **IntentRouter** → `RoutingDecision`（打 `METRIC routingDecision`）
3. **FailureDetector** + **EscalationScore**（检索前，可用 `riskLevel`）
4. 若需转人工 → **HandoffService** → 结束
5. **短路分支**（不进 Gate/Planner）：
   - 纯订单 → `answerOrderDeterministic`
   - 订单+规则 → `answerOrderWithPolicy`（检索 + LLM）
   - 纯工具 → `askWithTools`
6. **RetrievalGate** → 若 `shouldRetrieve`：
   - `rewriteQuestion`（LLM，结合 `chat:memory`）
   - `search()` 混合检索
   - **EvidenceSufficiency** 评估
   - 再次 Failure / Escalation
7. **Planner** → `ExecutionPlan` → `executePlan`
8. **finish**：写状态、写对话记忆、打 `METRIC ragTotalCostMs`

**关闭 Reliability：** `reliability.enabled=false` → 直接 `VectorService.askWithTools()`。

### 3.2 混合检索（`VectorService.search`）

```
用户问题 → 规范化
    → embedding（L1 Caffeine → L2 Redis → Ollama）
    → pgvector 相似度 Top30
    → ES BM25 Top30
    → RRF 融合
    → 去重、邻居扩展
    → rerankV1（关键词/短语/文件名主题加权）
    → Top5
```

- 向量失败：降级仅 BM25（`vector search failed, degrade to bm25 only`）
- 结果为空：`METRIC ragReject=below_threshold`

### 3.3 知识库构建

```
上传文件 → Tika 解析 → 按 docType 分块
    → 每块 embedding → INSERT document_chunk_vector
    → 同步索引 ES（Bm25SearchService）
```

- **版本**：`docId` + `version`，支持 `ACTIVE` / `EXPIRED`
- **回滚**：`POST /vector/rollback?docId=&version=`（指定版本，非仅上一版）
- **异步**：Kafka `knowledge-upload-topic` → Consumer 入库
- **并发**：Redisson 锁（按 chunk / 按文件）

## 4. Reliability 编排层

### 4.1 设计目标

- **可控**：不是每问都检索、都调 LLM
- **防幻觉**：证据不足/冲突时禁止生成
- **可降级**：LLM/向量/代理失败时有模板或确定性回复
- **可转人工**：多轮失败、高风险、用户明确要求

### 4.2 组件说明

| 组件 | 职责 |
|------|------|
| `IntentRouterService` | 规则意图路由，输出 `RoutingDecision` |
| `NeedRetrievalGateService` | 是否执行 hybrid search |
| `EvidenceSufficiencyService` | 检索结果是否足以作答、是否冲突 |
| `PlannerService` | 映射到 `AnswerMode` 与执行计划 |
| `ConversationFailureDetector` | 单次/会话失败信号 → 是否 handoff |
| `EscalationScoreService` | 跨轮升级分，阈值转人工 |
| `HandoffService` | 转人工话术 + 人工摘要 |
| `ConversationStateService` | 会话状态持久化与更新 |

### 4.3 AnswerMode

| Mode | 行为 | 是否调用 LLM |
|------|------|--------------|
| `RAG_HIGH_CONFIDENCE` | 证据充分，正常 RAG 生成 | 是 |
| `RAG_PARTIAL` | 部分证据，Prompt 要求标注不确定部分 | 是 |
| `RAG_INSUFFICIENT` | 模板：知识库依据不足 | 否 |
| `RAG_CONFLICT` | 模板：文档冲突，列出来源 | 否 |
| `NO_RAG` | 闲聊固定句 / 改写总结 | 视子意图 |
| `CLARIFY` | 请用户补充信息 | 否 |
| `REFUSE` | 拒答模板（Planner 有分支，Router 未使用） | 否 |
| `TOOL` | 工具/MCP 调用 | 否 |
| `HANDOFF` | 转人工 | 否 |

Orchestrator 短路时的日志 `answerMode`：`ORDER`、`ORDER_WITH_POLICY`、`TOOL` 等。

### 4.4 意图路由（IntentRouter）优先级

| 顺序 | path | 条件摘要 |
|------|------|----------|
| 1 | `clarify` | 空问题 |
| 2 | `handoff` | 赔偿、投诉、封号、律师、起诉、自动续费、乱扣费、盗刷 |
| 3 | `no_rag` | 改写/总结/翻译；整句你好/谢谢/在吗 |
| 4 | `tool` | 文件列表、版本+后缀、删除文件、任务状态 |
| 5 | `order` / `refund` | ORD 号或订单关键词；含退款/售后等 → `refund` |
| 6 | `clarify` | 长度≤8 且首轮会话 |
| 7 | `rag` | 默认 FAQ |

**riskLevel：** `high`（高风险词）/ `medium`（订单+规则）/ `low`（默认）。

### 4.5 证据充分性（Evidence）

**综合分：**

```
composite = 0.4×top1 + 0.3×query覆盖率 + 0.2×来源一致性 + 0.1×新鲜度
```

| 结果 | 条件 |
|------|------|
| 不足 | chunks 为空；top1 < 0.30；composite 过低 |
| 冲突 | Top 片段「支持/不支持」「可以/不可」矛盾 |
| partial | composite ≥ 0.45 等 |
| full | composite ≥ 0.65 且覆盖率 ≥ 0.35 |

### 4.6 失败检测与升级

**FailureDetector（硬规则，可立即 handoff）：**

- 用户：转人工、没用、你没回答…
- `userRepeatedQuestionCount >= 2`
- `routing.riskLevel == high`
- `lowConfidenceCount >= 3`（历史轮次，见下）
- 本次 evidence 不足 **且** `lowConfidenceCount >= 2`

**lowConfidenceCount：** 在 `finish()` → `recordTurn()` 中，当上一轮 `answerMode` 含 `INSUFFICIENT` 或 `PARTIAL` 时 +1；**检测发生在本轮 finish 之前**，故用的是历史累计。

**EscalationScore（软累计，默认阈值 8）：**

- 每轮先 `-escalationDecayPerTurn`（默认 1）
- 重复问 +2、failure +3、high risk +5、evidence 不足 +3、lowConfidence≥2 +2、含投诉/赔偿 +2
- `score >= threshold` → `handoffOffered=true`，状态 `ESCALATED`

### 4.7 会话状态机

| 状态 | 触发条件（简化） |
|------|------------------|
| `NORMAL` | 默认 |
| `CONFUSED` | 曾有低置信回答 |
| `FRUSTRATED` | 重复问≥2 或 lowConfidence≥2 |
| `ESCALATED` | 升级分达阈值或已 offer handoff |

**两套 Session 数据：**

| Redis Key | 用途 |
|-----------|------|
| `rag:state:{sessionId}` | 编排状态（Reliability） |
| `chat:memory:{sessionId}` | 多轮对话文本（改写/生成） |

## 5. 订单与工具路径

### 5.1 为什么不全走 LLM

- **订单**：字段来自 DB，确定性输出可审计、低延迟、LLM 不可用仍可服务
- **工具**：列文件/查版本等为结构化操作，无需生成

### 5.2 路径说明

| 场景 | 方法 | 检索 | LLM |
|------|------|------|-----|
| 纯订单（如 `ORD…` 查物流） | `answerOrderDeterministic` | 否 | 否 |
| 订单+规则（含退款/售后等词） | `answerOrderWithPolicy` | 是 | 是 |
| 知识库工具 | `handleToolQuestion` / MCP | 否 | 否 |

**订单+规则** 当前由 Orchestrator 短路，**不经过 Evidence 层**（已知改进点）。

### 5.3 订单规则关键词（`needRuleForOrderQuestion`）

能不能退、能退款、能退吗、可以退、可否退、怎么退、如何退、支持退、退款吗、退货吗、退款规则、退货规则、退款、退货、售后、赔偿、运费、超过、多久到账 等。

## 9. 降级与容错（Fallback）

```
用户请求
  ├─ reliability.enabled=false → 旧版 askWithTools
  ├─ DeepSeek 不可用 → callChatLlm fallback 模板
  ├─ 向量检索失败 → 仅 BM25
  ├─ 检索无结果 → below_threshold / RAG_INSUFFICIENT
  ├─ 证据冲突 → RAG_CONFLICT（不调 LLM）
  └─ 多轮失败 / 高风险 → HANDOFF
```

## 12. 已知限制与改进方向

1. **意图路由**为关键词规则，泛化依赖词表维护。
2. **`REFUSE` AnswerMode** 在 Planner 中存在，Router 未赋值，高风险走 `handoff`。
3. **订单+政策**短路跳过 Evidence，与 FAQ 链路不一致。
4. **MCP** 为 HTTP 工具风格，非标准 MCP 传输协议。
5. **赔偿/投诉** 在 `isHighRisk` 与 `needRuleForOrderQuestion`、Escalation +2 有重叠计分。
6. 证据冲突检测为启发式（支持/不支持等），可升级为 NLI 或二次检索。

## 14. 附录：与普通 RAG 面试总结

> 普通 RAG 解决「怎么找、怎么说」；本项目解决「**该不该找、能不能说、说错了怎么收**」。  
> 通过编排层把 **Router → Gate → Evidence → Planner → Handoff** 串成可观测、可降级、可转人工的生产链路，并用确定性订单/工具路径避开不必要的 LLM 幻觉风险。
